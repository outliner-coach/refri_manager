# Soul Document (`agent.md`)

## 1) 프로젝트 목적
- 이 프로젝트의 목적은 **회사 냉장고 운영을 AI로 자동화**하는 것이다.
- 핵심 목표:
  - 구성원이 음식 등록(사진, 소속/이름, 유통기한)을 쉽게 수행
  - 유통기한 임박/만료 알림을 자동으로 전달(Slack)
  - 관리자에게 등록/만료 현황을 투명하게 제공
  - 운영자의 반복 업무(확인, 리마인드, 기록)를 줄여 **AI 비용 지원 관리 및 운영 자동화**를 달성

## 2) 기본 폴더 구조 (권장 표준)
현재 루트가 비어 있으므로, 아래 구조를 기본 표준으로 사용한다.

```text
refri_manager/
├─ agent.md                # 이 Soul Document (항상 최초 읽기)
├─ PROGRESS.md             # 현재까지 완료된 작업 기록
├─ PLAN.md                 # 앞으로 해야 할 작업 목록
├─ README.md               # 프로젝트 개요, 실행 방법, 환경 변수
├─ commands/               # 운영 자동화 실행 커맨드 정의(YAML+Markdown)
├─ skills/                 # 업무별 전문 스킬 정의(SKILL.md + agents/openai.yaml)
├─ docs/                   # 기획/설계/운영 문서
│  ├─ architecture/        # 아키텍처, 시퀀스 다이어그램, ADR
│  ├─ api/                 # API 명세, 이벤트 계약
│  └─ operations/          # 운영 가이드, 장애 대응, 온보딩
├─ apps/
│  ├─ tablet-web/          # 냉장고 앞 태블릿 UI(음성, 카메라, 등록 플로우)
│  ├─ slack-bot/           # Slack 앱/봇(알림, 명령, 관리자 연동)
│  └─ admin-web/           # 관리자 대시보드(등록자/사진/유통기한 조회)
├─ services/
│  ├─ api/                 # 백엔드 API(인증, 등록, 조회, 관리)
│  ├─ worker/              # 스케줄/만료 스캔/알림 작업
│  └─ ai/                  # STT/파싱/이미지 처리 어댑터
├─ packages/               # 공통 모듈(타입, 유틸, 스키마)
├─ infra/                  # IaC, 배포, 모니터링, 시크릿 참조
├─ scripts/                # 개발/운영 자동화 스크립트
├─ tests/                  # 통합/E2E/성능/회귀 테스트
└─ data/
   ├─ samples/             # 샘플 데이터
   └─ migrations/          # DB 마이그레이션
```

## 3) Codex 기본 동작과 룰 (항상 준수)
- 작업 시작 전에 반드시 `agent.md`를 먼저 읽고, 목적/규칙/구조를 기준으로 판단한다.
- 설계 또는 구현 제안 시, 항상 아래 우선순위를 따른다:
  1. 운영 안정성(장애/복구/관측 가능성)
  2. 보안/개인정보 보호(최소 수집, 접근 통제)
  3. 비용 효율(토큰/모델/호출 횟수 최적화)
  4. 사용자 경험(태블릿 현장 흐름 최소 클릭)
- 기능 추가 시 필수 체크:
  - 로그/감사 추적 가능 여부
  - 실패 재시도/중복 방지(idempotency) 여부
  - 테스트 가능성(단위/통합) 여부
  - 운영 문서 업데이트 여부
- Slack/AI/카메라/음성 등 외부 연동은 “실패를 기본값”으로 설계한다.
- 데이터 모델은 만료 알림 자동화를 중심으로 설계한다(등록자, 음식, 사진, 유통기한, 상태).
- 애매한 요구가 있으면 임의 확장보다, 현재 목적(운영 자동화/비용 관리)에 맞는 최소 구현을 우선한다.

## 4) 시작 규칙 (중요)
- **앞으로 Codex가 이 폴더에서 작업을 시작할 때마다, 아래 순서로 항상 먼저 읽고 시작한다.**
  1. `agent.md`
  2. `PROGRESS.md`
  3. `PLAN.md`
- 이후 작업에서 판단 충돌이 생기면, 이 문서의 목적과 룰을 기준으로 의사결정한다.
- 새 작업에 착수하기 전, `PROGRESS.md`의 마지막 완료 항목과 `PLAN.md`의 최상단 미완료 항목을 기준으로 이어서 작업한다.
- 운영 자동화 작업(Command/Skill 추가·수정)일 경우, `commands/`와 `skills/` 변경사항을 함께 확인한다.

## 5) 유지보수 규칙
- 프로젝트 방향, 폴더 구조, 운영 원칙이 바뀌면 `agent.md`를 즉시 업데이트한다.
- 큰 결정(아키텍처/보안/비용 정책)이 확정되면 `docs/architecture` 또는 `docs/operations`에 상세를 남기고 `agent.md`에는 요약 원칙만 유지한다.
- 작업 완료 시 `PROGRESS.md`를 업데이트하고, 다음 할 일은 `PLAN.md` 상태를 갱신해 최신화한다.
